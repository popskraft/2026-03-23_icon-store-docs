## 1. Что это за система

Это не просто интернет-магазин. Это **операционная система каталога икон**, в центре которой находится единая карточка каждого товара, а BigCommerce и Amazon — выходные каналы публикации.

Главная инженерная задача — не сайт, а:
- надёжный приём и обработка изображений от разных людей
- создание канонических карточек из минимального набора фактов
- автогенерация текстов под каждый канал
- пакетная публикация и обновления без ручной работы

## 2. Роли в системе

Все роли — разные люди. Владелец ≠ Оператор ≠ Обработчик.

---

### Роль 1 — Автор / поставщик

| | |
|---|---|
| **Кто** | Иконописец, мастерская или оптовый поставщик |
| **Что делает** | Присылает исходные фото икон и первичные данные о товаре |
| **Входящее** | Запрос от владельца или инициатива поставщика |
| **Исходящее** | Фотографии (исходники) + данные о товаре в согласованном формате |
| **Инструменты** | Google Drive (папка incoming), email |
| **Ограничения** | Не имеет доступа к системе. Только передаёт материалы. |

---

### Роль 2 — Владелец проекта

| | |
|---|---|
| **Кто** | Владелец бизнеса, принимает стратегические и товарные решения |
| **Что делает** | Принимает материалы от поставщика, делает первичный approve, ведёт мастер-базу, принимает все решения по ассортименту и ценам |
| **Входящее** | Материалы от поставщика (фото + данные), вопросы от оператора |
| **Исходящее** | Заполненная мастер-база (Google Sheet) готовая к загрузке; подтверждение оператору что товар идёт в продажу |
| **Инструменты** | Google Drive, мастер-база (Google Sheet) |
| **Мастер-база** | Google Sheet в унифицированном формате: поля карточки товара заполнены так, чтобы оператор мог загрузить их в систему без ручного переформатирования. Источник истины до момента загрузки в PostgreSQL. |

**Первичный approve — зона ответственности владельца:**
- Все необходимые фото от поставщика получены и приняты
- Данные о товаре в мастер-базе полные и корректные
- Товар подтверждён к выводу в продажу (SKU, цена, описание источника)
- От поставщика больше ничего не требуется

> Approve публикации (контент + листинги) — зона ответственности **оператора**, не владельца.

---

### Роль 3 — Обработчик иллюстраций

| | |
|---|---|
| **Кто** | Штатный сотрудник, фрилансер с биржи или автоматический Python-скрипт — в зависимости от сценария запуска |
| **Что делает** | Получает исходные фото, нормализует геометрию, исправляет цвет и перспективу, удаляет фон, возвращает один обработанный мастер-файл |
| **Входящее** | Исходники в папке Google Drive: `incoming/{IC-000001}/` |
| **Исходящее** | Один approved master-файл в папке Google Drive: `approved/{IC-000001}/master.jpg` |
| **Инструменты** | Photoshop / Lightroom / аналоги; доступ только к своей папке в Drive |
| **Требования к результату** | Мин. 2000px по длинной стороне; белый или нейтральный фон; без обрезки объекта; RGB; JPG или PNG; предмет занимает ≥ 80% кадра |

**Три сценария (выбирается при запуске):**

| Сценарий | Когда применять | Доступ к Drive |
|----------|----------------|----------------|
| Python image worker (авто) | Стандартные фото с чистым фоном | Системный, через n8n |
| Штатный обработчик | Постоянный поток, контроль качества важен | Полный доступ к папке |
| Внешний аутсорс (биржа) | Разовые задачи, сложные случаи | Ограниченный: только папка конкретного SKU |

> Независимо от сценария — результат всегда в одном месте (`approved/{SKU}/master.jpg`) и с одинаковыми требованиями. Система не знает, кто обрабатывал.

### Роль 4 — Оператор магазина

| | |
|---|---|
| **Кто** | Операционный менеджер магазина, также отвечает за техническую инфраструктуру |
| **Что делает** | Курирует полный цикл от получения материалов до живых листингов; администрирует систему; вручную исправляет ошибки |
| **Инструменты** | Google Drive, n8n, PostgreSQL (SQL), BigCommerce Admin, Amazon Seller Central, VPS/сервер |

**Операционные обязанности оператора:**

| Зона | Конкретные действия |
|------|-------------------|
| Изображения | Забирает фото у обработчика; проверяет соответствие требованиям; загружает в Drive incoming/; при несоответствии возвращает на доработку |
| Каталог | Получает заполненную мастер-базу от владельца; запускает source-import в n8n; проверяет результат загрузки в PostgreSQL |
| Контент | Запускает генерацию (n8n); открывает review CSV; ставит APPROVE / REJECT / REGEN; загружает обратно |
| Публикация BC | Контролирует автоматическую прогрузку; вручную правит то, что встало некорректно в BigCommerce Admin |
| Публикация Amazon | Финальный approve перед публикацией листингов; проверяет листинги в SC после публикации; устраняет suppressed |
| Заказы | Принимает уведомления о FBM-заказах; отгружает; вводит трекинг |
| Инвентарь | Обновляет остатки в системе при пополнении склада |
| Мониторинг | Ежедневная проверка n8n Dashboard, алертов, ошибок батча |
| Система | Разворачивает и поддерживает VPS, PostgreSQL, n8n; управляет credentials; делает бэкапы |

**QA-чекпоинты оператора — контроль качества на каждом переходе:**

| # | Момент | Что проверяется |
|---|--------|----------------|
| ① | После получения от владельца | Все фото переданы, мастер-база заполнена полностью, данные соответствуют фото |
| ② | При передаче обработчику | ТЗ сформулировано чётко: формат, размер, требования к фону и кадрированию |
| ③ | После приёмки от обработчика | Соответствие ТЗ: размер ≥ 2000px, белый фон, предмет ≥ 80% кадра, без артефактов |
| ④ | Перед загрузкой в систему | Полнота мастер-базы: все поля заполнены, кол-во фото совпадает с кол-вом SKU |
| ⑤ | После публикации на сайте | Визуальная проверка карточки в BigCommerce: фото, описание, цена, категория |
| ⑥ | После публикации на Amazon | Визуальная проверка листинга в SC: главное фото, bullets, категория, статус active/suppressed |

> Оператор — единственная роль с доступом к PostgreSQL, n8n и серверной инфраструктуре.
> Оператор несёт ответственность за качество данных на каждом переходе — не только за нажатие кнопок.

### Роль 5 — Система (n8n + автоматика)

| | |
|---|---|
| **Кто** | Автоматические workflows: n8n, Python workers, Claude API, BC API, SP-API |
| **Что делает** | Выполняет все автоматизированные шаги между человеческими точками принятия решений |
| **Инструменты** | n8n, PostgreSQL, Claude API, Amazon SP-API, BigCommerce API v3, S3/R2, Remove.bg API |

**Что делает система (без участия человека):**
- Детектирует новые файлы в Drive → запускает image processing
- Обрабатывает изображения: удаление фона, нормализация, нарезка производных
- Генерирует контент через Claude API (ночные батчи)
- Публикует товары в BigCommerce после approve оператора
- Публикует листинги на Amazon после финального approve
- Синхронизирует остатки между каналами
- Опрашивает Amazon Orders API, создаёт заказы в PostgreSQL
- Отправляет алерты оператору при ошибках и низком стоке

> Система никогда не принимает решений о качестве контента и не публикует листинги без явного подтверждения человека.

## 3. Архитектурные принципы

1. **PostgreSQL — канонический реестр** — именно там живёт главная карточка иконы, а не в BigCommerce.
2. **BigCommerce — канал публикации** — принимает готовые данные, не является источником истины.
3. **Оригиналы неприкосновенны** — ни один входящий файл не перезаписывается автоматикой.
4. **Три стадии изображения** — Incoming → Approved Master → Published Derivatives. Смешивать нельзя.
5. **Каждый батч перезапускаем** — сбой не портит состояние, процесс возобновляется с последнего успешного шага.
6. **Человек подтверждает перед Amazon** — ни один LLM-текст не уходит в листинг без явного approve.
7. **Каналы изолированы** — текст для Amazon и текст для сайта генерируются независимо из одних исходных фактов.
8. **Цена живёт в мастер-карточке** — не в BigCommerce напрямую. Изменение правила → пакетное обновление всех каналов.
9. **Минимум технологий** — каждый компонент должен быть заменим без переписывания всего.
10. **Рабочая документация на русском** — финальные артефакты для публикации на английском.

## 5. Центральный объект: Icon Master Record

Каноническая карточка иконы — главный объект всей системы. Из неё рождаются тексты, изображения, цены и публикации. Живёт в PostgreSQL. Полная DDL — в **Spec-1 (раздел 15)**.

### 5.1 Идентичность

```
icon_id           — внутренний постоянный UUID
merchant_sku      — главный идентификатор (IC-000001, IC-000001-S/M/L)
supplier_sku      — артикул поставщика (внешний reference, не PK)
product_family_id — если есть вариации одной иконы
asin              — после публикации на Amazon
bc_product_id     — после публикации в BigCommerce
```

**SKU-правило:** `IC-000001` — простой последовательный номер. Семантику в SKU не кодируем — она живёт в полях карточки. Суффикс `-S/M/L` только для размерных вариантов.

### 5.2 Исходные факты (от поставщика)

```
title_raw           — исходное название от автора
saint_or_subject    — кто изображён
feast               — праздник (если применимо)
purpose             — назначение (домашняя молитва, путевая, подарок...)
style               — стиль
iconographic_type   — иконографический тип
material            — материал (дерево, металл...)
technique           — техника (печать, ручная работа...)
construction_type   — конструкция (плоская доска, триптих, складень...)
origin_school       — школа / происхождение
size_w_mm, size_h_mm, size_d_mm
weight_g
manufacturer        — автор / мастерская
notes               — свободные заметки
```

### 5.3 Изображения

```
incoming_source_uri     — путь к исходнику (Google Drive path или URL)
approved_master_uri     — путь к approved master в S3/R2
derivatives             — JSONB: {amazon-main, site-pdp, ...} → URL
image_qa_status         — incoming | processing | approved | rejected
image_qa_notes
last_processed_at
```

### 5.4 Контент

```
content_version
content_status          — not_started | generating | review | approved | published
approved_by
approved_at
review_notes

site_title_en
site_description_html_en
meta_title_en
meta_description_en

amazon_title            [★ sync]
amazon_bullet_1..5      [★ sync]
amazon_description_html
amazon_backend_search_terms [★ sync]
amazon_browse_node      [★ sync]
amazon_item_type_keyword [★ sync]
amazon_target_marketplace [★ sync]
```

`[★ sync]` — поля, которые должны совпадать в карточке, BigCommerce metafields и Amazon payload.

### 5.5 Цены

```
base_cost           — закупочная цена
pricing_rule_id     — ссылка на правило расчёта маржи
site_price          — рассчитывается: base_cost × markup
amazon_price        — рассчитывается: site_price × amazon_factor
map_price           — минимальная допустимая цена
price_last_updated
```

### 5.6 Публикация

```
bc_status           — draft | published | archived
amazon_status       — not_listed | pending | live | suppressed | live_manual
publication_notes
```

### 5.7 Inventory

```
site_stock          — остаток для сайта (отдельный пул)
amazon_reserved     — остаток зарезервирован под Amazon FBM (отдельный пул)
reorder_threshold
supplier_lead_days
```

### 5.8 Sync / Audit

```
last_hash           — SHA-256 от значимых полей карточки
last_synced_at
batch_id
sync_error_code
source_of_last_update
```

### Пайплайн A — Приём каталога

#### Как получить доступ к интерфейсу n8n

n8n — это веб-приложение с визуальным интерфейсом. Оператор работает с ним через обычный браузер.

**Адрес для входа:**

| Вариант деплоя | Адрес |
|----------------|-------|
| VPS с настроенным доменом | `https://n8n.your-domain.com` (домен задаётся при установке) |
| VPS без домена (временный доступ) | `http://YOUR_VPS_IP:5678` (заменить на реальный IP сервера) |
| n8n Cloud | `https://app.n8n.cloud` → войти под аккаунтом |

> Конкретный URL задаётся при деплое сервера и сообщается оператору системным администратором или DevOps. Если адрес неизвестен — запросить у того, кто устанавливал n8n.

**Первый вход:**
1. Открыть адрес в браузере
2. Ввести email и пароль n8n-аккаунта (задаются при установке)
3. Откроется дашборд со списком всех workflows

> ⚠️ n8n не открывается через мобильный браузер — используй компьютер. При работе через VPS без HTTPS браузер может показать предупреждение — его можно проигнорировать в рабочих условиях, но для продакшна нужен SSL-сертификат (Let's Encrypt, настраивается при деплое).

#### Как запускается

Три варианта — выбирается при настройке системы:

| Вариант | Как работает | Рекомендация |
|---------|-------------|--------------|
| **Вручную оператором** | Оператор открывает n8n → нажимает Run на workflow `source-import` → указывает файл | ✅ Для MVP — полный контроль |
| **Автоматически по файлу** | n8n следит за папкой Google Drive; когда появляется новый файл — запускается сам | Удобно при потоке от нескольких поставщиков |
| **По расписанию** | n8n проверяет папку раз в день в заданное время | Если файлы кладутся нерегулярно |

#### Формат мастер-базы (Google Sheet)

Мастер-база — это Google Sheet который заполняет владелец. Каждая строка = один SKU. Колонки должны строго совпадать с именами ниже, иначе n8n не распознает данные.

**Обязательные поля** (без них строка отклоняется):

| Колонка | Пример | Описание |
|---------|--------|----------|
| `saint_or_subject` | Mother of God | Кто изображён |
| `material` | Wood | Материал основы |
| `technique` | Print | Техника исполнения |
| `construction_type` | flat-board | Тип конструкции |
| `size_w_mm` | 130 | Ширина в мм |
| `size_h_mm` | 180 | Высота в мм |
| `base_cost` | 12.50 | Закупочная цена в USD |

**Необязательные поля** (загружаются если есть):

| Колонка | Пример |
|---------|--------|

#### Сценарии ошибок валидации

Принцип: **плохая строка не убивает весь файл**. n8n обрабатывает строки поштучно.

| Ошибка | Что происходит | Действие оператора |
|--------|---------------|-------------------|
| Пустое обязательное поле | Строка пропускается, ошибка логируется | Заполнить в мастер-базе, перезапустить только эту строку |
| Дубль SKU в файле | Первая строка принята, дубль пропущен | Проверить какая версия верная |
| SKU уже есть в PostgreSQL | Пропускается (skip, не перезатирает) | Если нужно обновить — PATCH-батч отдельно |
| Нечисловое значение в размере | Строка пропускается | Исправить значение в файле |
| Неизвестный материал | Загружается как есть, флаг `REVIEW_MATERIAL` | Проверить вручную и обновить |
| Отрицательная или нулевая цена | Строка пропускается | Указать корректную закупочную цену |

#### Как генерируется merchant_sku

SKU — уникальный идентификатор каждого товара. Формат: `IC-000001`, `IC-000002`, и т.д. Это внутренний артикул, он не меняется после создания и используется везде: в PostgreSQL, в BigCommerce, в Amazon.

**Рекомендуемый вариант — автоматическая генерация через PostgreSQL**

PostgreSQL умеет автоматически назначать следующий номер по порядку через механизм «sequence» (последовательность). Оператор и владелец не придумывают SKU вручную — они появляются сами при загрузке мастер-базы.

При настройке базы данных (один раз при деплое) создаётся:
```sql
-- Счётчик — автоматически увеличивается при каждой новой записи
CREATE SEQUENCE icons_sku_seq START 1;

-- Функция, формирующая SKU из счётчика
CREATE OR REPLACE FUNCTION gen_merchant_sku()
RETURNS TEXT AS $$
  SELECT 'IC-' || LPAD(nextval('icons_sku_seq')::TEXT, 6, '0');
$$ LANGUAGE SQL;
```

После этого первый загруженный товар получает `IC-000001`, второй — `IC-000002`, и так далее. Никаких ручных действий.

**Альтернативный вариант — владелец задаёт SKU вручную**

Если нужен контроль над нумерацией (например, привязка к артикулу поставщика), добавить колонку `merchant_sku` в мастер-базу как обязательное поле. n8n возьмёт значение из файла и не будет генерировать автоматически.

> Для MVP рекомендуется автогенерация. Владелец не заполняет SKU — они создаются системой.

#### Что происходит после успешного импорта

После загрузки строк в PostgreSQL Pipeline A завершён. Следующий шаг — за оператором.

**Что делает оператор:**
1. Читает отчёт n8n: сколько загружено, сколько пропущено и почему
2. Проверяет пропущенные строки — при необходимости просит владельца исправить в мастер-базе и перезапускает импорт только для этих строк
3. Проверяет что для новых SKU фото уже лежат в Google Drive в папке `incoming/{SKU}/`
   - Если фото нет — уведомляет обработчика или просит владельца
   - Если фото есть — переходит к Pipeline B (чекпоинт ③)

> **Важно:** Pipeline B не запускается автоматически после A. Это намеренно — оператор сначала убеждается что данные корректны, и только потом даёт старт обработке фотографий.

#### Как обновить уже существующий SKU

Если товар уже есть в системе и нужно изменить данные (поправить цену, описание, вес), нельзя просто снова загрузить его через `source-import` — строка будет пропущена (защита от случайной перезаписи).

Для обновлений используется отдельный workflow `source-patch` в n8n.

**Пошаговая инструкция:**

**Шаг 1 — Подготовить файл обновлений:**
- Скопировать шаблон мастер-базы
- Оставить только строки с теми SKU, которые нужно обновить
- Заполнить только изменяющиеся поля (остальные оставить пустыми — они не перезатрутся)
- Колонку `merchant_sku` заполнить обязательно — это ключ для поиска в базе

**Шаг 2 — В n8n:** открыть workflow `source-patch` → нажать Run → указать файл.

Workflow найдёт каждый SKU в PostgreSQL и обновит только непустые поля.

**Шаг 3 — Проверить отчёт:**
```
source-patch завершён: 5 обновлено / 0 пропущено
  Обновлены поля: base_cost, notes (для IC-000015..IC-000019)
```

**Шаг 4 — Если изменились данные влияющие на контент** (название иконы, техника, материал) — запустить `content-generate` заново для этих SKU. Они получат `content_version = 2` и уйдут на повторный review к оператору.

> **Цены** обновляются отдельно через workflow `price-update` (раздел 8) — он пересчитывает по правилам и синхронизирует сразу в оба канала.

#### Примечание: учётные данные Google — одни на оба пайплайна

Credentials, созданные в шагах выше (Google Sheets API — Service Account), **повторно используются в Pipeline B** без каких-либо изменений.

В Pipeline B используется триггер **Google Drive Trigger** — он следит за папкой `incoming/` и реагирует на появление новых файлов. Этот триггер использует те же Google Drive API права, которые уже включены в шаге «Шаг 2 — Включить нужные API» (Google Drive API был активирован там специально для этой цели).

**Что это означает на практике:**
- Отдельно настраивать Google Drive credentials для Pipeline B **не нужно**
- В ноде Google Drive Trigger → поле Credential → выбрать `Google Sheets (service account)` — это то же подключение
- Если при тестировании Pipeline B появляется ошибка доступа к Drive — проверить что Service Account email добавлен как Viewer в саму папку `incoming/` в Google Drive (аналогично тому, как его добавляли к мастер-базе)

### Пайплайн B — Обработка изображений

#### Кто делает первичную сортировку

Сортировка происходит в два этапа разными людьми:

**Этап 1 — Владелец** (при приёме от поставщика):
- Смотрит фото от поставщика
- Отделяет явно непригодные (смазанные, тёмные, неправильный ракурс)
- Раскладывает по папкам Drive: `incoming/{IC-000001}/` — только отобранные фото

**Этап 2 — Оператор** (при приёме от владельца):
- Проверяет каждую папку по техническим критериям QA (чекпоинт ③)
- Если фото не проходит: создаёт ТЗ для обработчика с конкретными правками
- Только после одобрения оператора система начинает обработку

#### Сколько фото нужно на один SKU

Анализ основан на требованиях Amazon (до 9 фото), требованиях BigCommerce (рекомендуется 3+) и типах конструкций икон:

| Тип иконы | Минимум (Amazon) | Оптимум | Максимум полезный |
|-----------|:---:|:---:|:---:|
| Плоская доска — печать | 2 (front + detail) | 4 | 6 |
| Плоская доска — ручная работа | 2 (front + detail) | 5 | 7 |
| Металлическая риза | 2 (front + detail-riza) | 4 | 6 |
| Триптих / складень | 2 (открыт + закрыт сбоку) | 4 | 6 |
| Путевая икона | 3 (front + back + side) | 5 | 7 |

**Рекомендуемый минимум для старта: 2 фото на SKU** — front + detail. Этого достаточно для публикации на Amazon. Сайт работает и с одним front.

**Оптимальный набор (4–5 фото):**
1. Front — главное фото, белый фон, предмет на весь кадр
2. Detail — крупный план лика или характерной детали
3. Back / Side — обратная сторона или профиль (показывает материал, толщину)
4. Size reference — рядом с рукой или монетой для масштаба
5. Lifestyle — икона на полке или в интерьере (для сайта особенно ценно)

**Вывод:** при первом запуске собирайте минимум 2 фото на каждый SKU. Для листинга Amazon это покрывает требования. Lifestyle-фото можно добавлять постепенно не останавливая запуск.

#### Три стадии обработки — строго разделены

```
Stage A: Incoming (человек → Drive)
Владелец сортирует фото от поставщика → папки incoming/{SKU}/
    ↓
Оператор проверяет по QA-критериям:
    ├── OK → оставить как есть, Drive webhook запускает Stage B
    └── Проблемные → ТЗ обработчику → обработчик возвращает в approved/{SKU}/
                                       → оператор принимает → Stage B

Stage B: Approved Master (Python worker → S3/R2)
    - удаление фона: Remove.bg API → fallback: rembg (локальная модель)
    - нормализация яркости, контраста, баланса белого
    - проверка: размер ≥ 2000px по длинной стороне
    - сохранение: icons/originals/{IC-000001}/master.jpg в S3/R2
    - PostgreSQL: approved_master_uri обновлён, image_qa_status = approved

Stage C: Published Derivatives (Python worker → S3/R2, авто из Stage B)
    - amazon-main:    2000×2000, белый фон, JPG, q95
    - amazon-gallery: 1500×1500, белый фон, JPG, q90
    - site-pdp:       800×800,   белый фон, WebP, q85
    - site-card:      400×400,   белый фон, WebP, q80
    - site-zoom:      1600×1600, белый фон, JPG, q90
    - thumbnail:      150×150,   fill crop, WebP, q70
    Сохранение: icons/derived/{IC-000001}/{variant}.ext
    PostgreSQL: поле derivatives{} заполнено URL-адресами всех версий
```

**Матрица обязательных ракурсов:**

| Тип конструкции | Минимум для сайта | Минимум для Amazon |
|-----------------|:-----------------:|:------------------:|
| Плоская доска (печать/роспись) | front | front + detail-frame |
| Металлическая риза | front | front + detail-riza |
| Триптих / складень | front (открыт) | front (открыт) + side (закрыт) |
| Путевая икона | front | front + back + side |

`media_partial` = front есть, остальное нет → сайт OK, Amazon заблокирован до добавления фото.
`media_blocked` = front отсутствует → все шаги заблокированы для этого SKU.

#### Как настроить Google Drive Trigger в n8n — одноразовая настройка

Drive Trigger — это механизм при помощи которого n8n узнаёт что в папке `incoming/` появились новые фото. Без него n8n не будет знать о новых файлах.

**Рекомендуемый вариант — Google Drive Trigger нод в n8n**

n8n имеет встроенный нод «Google Drive Trigger». Он периодически опрашивает Drive на наличие изменений (каждые 1–5 минут) и при появлении нового файла автоматически запускает workflow.

**Как настроить:**
1. В workflow `image-process` первый нод — «Google Drive Trigger»
2. **Credential:** те же Google Drive credentials (Service Account) которые настроены в Pipeline A
3. **Resource:** File → **Event:** File Created
4. **Папка:** указать ID папки `incoming/` в Drive
   - ID виден в URL: `drive.google.com/drive/folders/**ВОТ_ЭТО**`
5. **Poll Time:** `Every 5 Minutes` (для старта достаточно)
6. Нажать **Active** переключатель на workflow — запускает мониторинг

После активации n8n будет автоматически реагировать на новые файлы в папке.

**Альтернатива для MVP — ручной запуск**

Если автотриггер не нужен на старте, оператор открывает n8n → workflow `image-process` → нажимает «Run manually» → указывает SKU или «обработать все новые» (где `image_qa_status = 'incoming'`).

> Рекомендация: начать с ручного запуска. Подключить автотриггер когда поток товаров станет регулярным.

#### Как n8n вызывает Python worker

Python worker — скрипт обработки фото, который работает на том же VPS-сервере что и n8n. Он всегда запущен и ждёт задач. n8n обращается к нему через HTTP-запрос — как к мини-веб-сервису.

**Схема взаимодействия:**

```
n8n workflow (оркестратор)
    ↓ POST /process (HTTP Request нод)
Python worker FastAPI (порт 8001 на том же сервере)
    ↓ скачивает фото из Drive, удаляет фон, нарезает производные, загружает в S3/R2
    ↑ возвращает JSON с URL всех готовых файлов
n8n
    ↓ записывает URL в PostgreSQL
```

**Пример запроса который отправляет n8n:**
```json
POST http://localhost:8001/process
{
  "sku": "IC-000001",
  "source_uri": "https://drive.google.com/file/d/ABC123/view",
  "output_bucket": "r2://icons/"
}
```

**Пример ответа который приходит обратно:**
```json
{
  "status": "ok",
  "master_uri": "https://r2.example.com/icons/originals/IC-000001/master.jpg",
  "derivatives": {
    "amazon-main":    "https://r2.example.com/icons/derived/IC-000001/amazon-main.jpg",
    "amazon-gallery": "https://r2.example.com/icons/derived/IC-000001/amazon-gallery.jpg",
    "site-pdp":       "https://r2.example.com/icons/derived/IC-000001/site-pdp.webp",
    "site-card":      "https://r2.example.com/icons/derived/IC-000001/site-card.webp",
    "site-zoom":      "https://r2.example.com/icons/derived/IC-000001/site-zoom.jpg",
    "thumbnail":      "https://r2.example.com/icons/derived/IC-000001/thumbnail.webp"
  }
}
```

n8n берёт поле `derivatives` и записывает URL в поле `derivatives_json` в PostgreSQL. После этого все пайплайны знают где лежат фото.

**Нод в n8n:** «HTTP Request» → Method: POST → URL: `http://localhost:8001/process` → Body: JSON с полями выше. Настройка worker (установка FastAPI, зависимостей Python) выполняется при деплое VPS — по инструкции из Spec-2 (раздел 16).

#### Что происходит при двойном сбое обработки фона

Remove.bg — внешний платный сервис. rembg — локальная модель на сервере. Если оба не справляются:

| Ситуация | Что делает система | Что получает оператор |
|----------|-------------------|-----------------------|
| Remove.bg недоступен (API error) | Автоматически переключается на rembg | Ничего — переключение прозрачное |
| rembg дал плохой результат (confidence < 0.85) | SKU получает флаг `REVIEW_BG_FAILED` в `sync_error_code` | Уведомление с SKU и причиной |
| Оба полностью упали (критическая ошибка сервера) | `image_qa_status` остаётся `processing`, `sync_error_code = 'BG_REMOVAL_FAILED'` | Уведомление |

**Что делает оператор при флаге `BG_REMOVAL_FAILED`:**

Посмотреть оригинальное фото в Drive (`incoming/{SKU}/`) и выбрать один из трёх путей:

**Путь 1 — Фон уже белый или почти чистый.** Удаление фона не нужно. Выполнить в PostgreSQL:
```sql
UPDATE icons
SET sync_error_code = NULL,
    image_qa_status = 'approved'
WHERE merchant_sku = 'IC-000001';
```
Затем перезапустить `image-process` для этого SKU — worker пропустит шаг удаления фона.

**Путь 2 — Фон сложный, нужна ручная обработка.** Отправить фото обработчику с ТЗ (шаблон ниже).

**Путь 3 — Фото низкого качества.** Отклонить SKU (`image_qa_status = 'rejected'`), запросить у владельца новое фото от поставщика.

#### Шаблон ТЗ для обработчика изображений

Когда оператор видит что фото требует ручной работы, он создаёт ТЗ. Формат — текстовый файл, кладётся в Drive в папку `incoming/{SKU}/ТЗ.txt`.

```
SKU: IC-000001
Исходное фото: incoming/IC-000001/photo_01.jpg

ЗАДАЧИ (отметить нужное):
[ ] Удалить фон → заменить на белый (#FFFFFF)
[ ] Исправить геометрию (перспектива, наклон)
[ ] Нормализовать яркость и контраст
[ ] Обрезать: предмет должен занимать ≥ 80% кадра
[ ] Убрать блики / отражения
[ ] Другое: _______________

ТРЕБОВАНИЯ К РЕЗУЛЬТАТУ:
- Минимальный размер: 2000px по длинной стороне
- Формат: JPG или PNG, цветовое пространство RGB
- Фон: чисто белый (#FFFFFF)
- Предмет не обрезан, нет посторонних объектов в кадре
- Файл назвать: master.jpg

КУДА ПОЛОЖИТЬ ГОТОВЫЙ ФАЙЛ:
approved/IC-000001/master.jpg

СРОК: _______________
ПРИМЕЧАНИЯ: _______________
```

**После того как обработчик положил готовый файл:**

1. Оператор проверяет результат (чекпоинт ③) — открывает файл в Drive
2. Если принято: обновить статус в PostgreSQL:
   ```sql
   UPDATE icons
   SET image_qa_status = 'approved'
   WHERE merchant_sku = 'IC-000001';
   ```
3. Запустить `image-process` вручную для этого SKU — Python worker выполнит Stage B (создаст master в S3/R2) и Stage C (нарежет производные)
4. Если отклонено: вернуть ТЗ обработчику с пометкой что именно не так

#### Что происходит если S3/R2 недоступен при загрузке

После того как Python worker обработал фото, последний шаг — загрузка файлов в S3 / Cloudflare R2. Если в этот момент хранилище недоступно (сетевой сбой, ошибка авторизации, бакет превысил квоту), возникает ошибка загрузки.

**Как это проявляется:**

Python worker возвращает в n8n ответ с ошибкой вместо JSON с URL:
```json
{
  "status": "error",
  "stage": "upload",
  "error": "S3 connection timeout / AccessDenied / Bucket quota exceeded",
  "sku": "IC-000001"
}
```

n8n получает этот ответ, не находит поля `derivatives` и записывает в PostgreSQL:
```
image_qa_status = 'processing'  (не меняется на 'approved')
sync_error_code = 'STORAGE_UPLOAD_FAILED'
```

**Что делает оператор при ошибке `STORAGE_UPLOAD_FAILED`:**

Шаг 1 — Проверить доступность хранилища:
- Открыть Cloudflare Dashboard (или AWS S3 Console) → убедиться что бакет существует и API-ключ активен
- Частая причина: истёк API Token R2 или изменились permissions — нужно перевыпустить токен и обновить credentials в n8n (Credentials → найти R2/S3 credential → обновить ключ)

Шаг 2 — После устранения причины — перезапустить обработку:
```
n8n → workflow image-process → Run для SKU IC-000001
```
Worker скачает фото из Drive заново и повторит загрузку — предыдущая обработанная версия не сохранилась локально на сервере.

Шаг 3 — Проверить результат:
```sql
SELECT image_qa_status, sync_error_code
FROM icons
WHERE merchant_sku = 'IC-000001';
-- Ожидается: image_qa_status = 'approved', sync_error_code = NULL
```

> **Важно:** Python worker не хранит промежуточные обработанные файлы на диске VPS после сбоя загрузки. Вся обработка (удаление фона, нарезка) выполнится заново при следующем запуске. Это нормально — операция идемпотентна (результат одинаков при любом числе повторений).

### Пайплайн C — Генерация контента

#### Условие запуска (исправлено)

Карточка готова к генерации контента когда выполнены оба условия в PostgreSQL:
```sql
image_qa_status = 'approved'   -- фото прошли обработку
AND content_status = 'not_started'  -- контент ещё не генерировался
```

Поле `source_ready` не используется — это было ошибкой в предыдущей версии документа.

#### Как и когда запускается батч

«Ночной батч» — неточное название. Никакой круглосуточной работы нет. Это **плановый батч**, который оператор запускает удобное рабочее время. Два варианта:

**Вариант 1 — Ручной запуск (рекомендуется для MVP):**
- Оператор видит что накопились SKU с `image_qa_status = approved`
- Открывает n8n → запускает workflow `content-generate`
- n8n отбирает все готовые карточки, отправляет в Claude API
- Через несколько минут (или часов для большого батча) — review CSV готов
- Оператор получает уведомление и делает review в удобное время

**Вариант 2 — По расписанию:**
- n8n запускает `content-generate` автоматически, например каждый понедельник в 9:00
- Отбирает все готовые SKU, генерирует, кладёт CSV в Drive
- Оператор открывает Drive в начале недели и видит готовый review пакет
- Удобно когда поток товаров регулярный

**Размер батча:**
- ≤ 10 SKU → Claude Messages API (синхронно, результат сразу)
- > 10 SKU → Claude Message Batches API (асинхронно, результат через 15–60 мин, стоимость в 2 раза ниже)

#### Что происходит если Claude API недоступен

Claude API — внешний сервис, иногда бывают технические сбои. Сценарии:

| Ситуация | Что делает n8n | Действие оператора |
|----------|---------------|-------------------|
| Временный сбой (минуты) | Retry 3 раза с паузой 5 мин между попытками | Ничего — n8n справится сам |
| Сбой продолжается > 15 мин | Батч переходит в `FAILED_WITH_EXCEPTIONS` | Уведомление оператору. Проверить status.anthropic.com, повторить позже |
| Rate limit превышен | n8n ждёт и продолжает | Ничего — автоматически |
| Частичный результат (10 из 20 SKU) | Успешные записаны, неудачные в статусе FAILED | Retry только неудачных через n8n |

Важно: SKU с успешно сгенерированным контентом **не теряются** при сбое. При retry обрабатываются только те что не прошли.

#### Полный поток

```
n8n отбирает карточки:
    image_qa_status = 'approved' AND content_status = 'not_started'
    ↓
Для каждого SKU: формирует prompt input JSON
    (source facts + глоссарий + правила каналов)
    ↓
≤ 10 SKU → Claude Messages API (результат сразу)
> 10 SKU → Claude Message Batches API (результат через 15–60 мин)
    ↓
n8n получает ответы → валидирует каждый:
    - все обязательные поля заполнены?
    - длина amazon_title ≤ 200 символов?
    - backend_search_terms ≤ 249 байт?
    - нет prohibited phrases?
    ↓
Генерирует review CSV + JSONL архив → кладёт в Google Drive reviews/batch_{дата}/
    ↓
Уведомляет оператора (email или n8n notification) — включая прямую ссылку на папку
    ↓
Оператор: APPROVE / REJECT / REGEN в CSV
    ↓
APPROVE → контент записывается в PostgreSQL + BC metafields
          content_status = 'approved'
          content_versions: новая запись с полным JSON
```

**Output contract (JSON на каждый SKU):**
```json
{
  "sku": "IC-000001",
  "amazon": {
    "title": "",
    "bullet_1": "", "bullet_2": "", "bullet_3": "", "bullet_4": "", "bullet_5": "",
    "description_html": "",
    "backend_search_terms": "",
    "browse_node": "",
    "item_type_keyword": "",
    "target_marketplace": "US"
  },
  "site_en": {
    "title": "",
    "description_html": "",
    "meta_title": "",
    "meta_description": ""
  },
  "seo": { "primary_keyword": "", "secondary_keywords": [] },
  "compliance_flags": [],
  "review_notes": "",
  "content_version": 1,
  "generated_at": ""
}
```

#### Как физически работать с review CSV

После генерации n8n кладёт CSV-файл в Google Drive в папку `reviews/batch_{дата}/review.csv`. Оператор работает с этим файлом.

**Как оператор узнаёт что CSV готов:**

n8n отправляет уведомление (email или встроенный n8n notification) следующего вида:
```
Тема: Review CSV готов — batch_2026-03-24 (12 SKU)

Контент для 12 SKU сгенерирован и готов к review.

📂 Папка в Google Drive:
https://drive.google.com/drive/folders/FOLDER_ID_ЗДЕСЬ

Файл: reviews/batch_2026-03-24/review.csv
Откройте ссылку → отредактируйте колонку L (APPROVE / REJECT / REGEN) → сохраните.
```

> Конкретная ссылка формируется автоматически n8n при создании папки. FOLDER_ID — это ID папки в Google Drive, n8n получает его из ответа Google Drive API после создания файла и подставляет в уведомление.

**Пошаговая инструкция:**

**Шаг 1.** Открыть ссылку из уведомления (или вручную: Google Drive → папка `reviews/` → найти папку с последней датой).

**Шаг 2.** Кликнуть на `review.csv` — Drive автоматически откроет его как Google Sheets (редактировать онлайн, не скачивая).

**Шаг 3.** В таблице есть колонки:

| Колонка | Содержание |
|---------|-----------|
| A — `merchant_sku` | Артикул товара |
| B — `amazon_title` | Заголовок для Amazon |
| C–G — `bullet_1..bullet_5` | Пять буллетов |
| H — `backend_search_terms` | Скрытые ключевые слова |
| I — `browse_node` | Числовой ID категории Amazon |
| J — `site_title` | Заголовок для сайта |
| K — `site_description` | Описание для сайта |
| **L — `decision`** | **Сюда оператор пишет решение** |
| **M — `review_notes`** | **Сюда — причина или пожелания** |

**Шаг 4.** Для каждой строки заполнить колонку L одним из трёх значений: `APPROVE`, `REJECT`, или `REGEN`.

**Шаг 5.** Сохранить (Ctrl+S — в Google Sheets сохраняется автоматически при каждом изменении).

**Как n8n узнаёт о решениях оператора — два варианта:**

| Вариант | Как работает | Рекомендация |
|---------|-------------|--------------|
| **Watch folder (автоматически)** | n8n проверяет файл каждые 15 минут. Как только колонка L заполнена — применяет решения сразу | ✅ Удобнее всего |
| **Ручной запуск** | Оператор заходит в n8n → запускает workflow `content-review-apply` | Полный контроль, лишний шаг |

> Можно работать с CSV в несколько заходов: заполнить часть строк сегодня, часть — завтра. n8n применяет только те строки, где заполнена колонка L.

#### Что происходит с REJECT и REGEN — объяснение трёх решений

**APPROVE** — контент принят. n8n записывает его в PostgreSQL, устанавливает `content_status = 'approved'`. SKU становится готов к Pipeline D (публикация в BigCommerce).

**REJECT** — контент отклонён полностью, товар возвращается в начало.
- `content_status` → `not_started`
- Поля контента в PostgreSQL очищаются
- Причина из `review_notes` сохраняется в `publication_notes` для истории

Когда использовать REJECT: если данные карточки товара настолько неполные или ошибочные что результат генерации в принципе был бесполезным. Например, не заполнен `saint_or_subject`, или `technique` указана неверно.

После REJECT оператор или владелец дополняет данные в PostgreSQL (через `source-patch`), после чего SKU снова попадёт в следующий запуск `content-generate`.

**REGEN** — запросить повторную генерацию с замечаниями.
- `content_status` → `not_started`
- Замечания из колонки M (`review_notes`) сохраняются и передаются Claude при следующей генерации

Когда использовать REGEN: контент понятен, но не совсем то что нужно. Например: не тот тон, упущен важный акцент, нужно другое ключевое слово.

Примеры замечаний для REGEN в колонке M:
```
"Emphasize handpainted technique with egg tempera. Do not use the word 'print'."
"Add feast day July 21 to description. Mention Russian iconographic tradition."
"Title too generic. Must include 'Orthodox' and material 'wood'."
```

#### Как review_notes попадают обратно при REGEN

При следующем запуске `content-generate` n8n находит SKU с `content_status = 'not_started'` и непустым `review_notes`. Для таких SKU он добавляет замечания в промпт Claude как дополнительные инструкции.

Структура промпта при REGEN:
```
[Системный промпт — правила, глоссарий]
[Данные карточки товара — факты]
REVISION NOTES FROM OPERATOR: "Emphasize handpainted technique..."
Please revise the content taking into account the notes above.
```

После успешной REGEN поле `review_notes` очищается, `content_version` увеличивается на 1 (например, становится 2). Оператор снова видит этот SKU в следующем review CSV.

> **Совет:** пишите замечания конкретно и по-английски — Claude получает их напрямую. Расплывчатые комментарии типа «сделать лучше» не дадут нужного результата.

#### Когда становится доступен Pipeline D

Pipeline D (публикация в BigCommerce) запускается когда оператор поставил APPROVE и n8n применил решение.

**Условие в PostgreSQL:**
```sql
content_status = 'approved'
```

**Два варианта когда запускается D:**

**Вариант 1 — Автоматически (рекомендуется):** n8n следит за изменениями `content_status`. Как только появляется `approved` — автоматически запускает `bc-publish` для этого SKU. Товар появляется в BigCommerce в статусе Draft (не виден покупателям) без участия оператора.

**Вариант 2 — Вручную:** Оператор отдельно заходит в n8n и запускает `bc-publish`. Используется если нужно опубликовать группу товаров одновременно, а не по одному.

> После публикации в BC товар в статусе **Draft** — не виден покупателям. Перевод в Visible описан в Pipeline D → «Draft → Visible».

### Пайплайн D — Публикация в BigCommerce

#### Предварительная настройка: как создать API Token в BigCommerce

n8n обращается к BigCommerce через API — это требует API Token. Токен создаётся один раз при первом деплое. Без него n8n не сможет создавать или обновлять товары.

**Пошаговая инструкция:**

**Шаг 1 — Открыть BigCommerce Admin:**
1. Войти в `https://store-{STORE_HASH}.mybigcommerce.com/manage/` (или через `login.bigcommerce.com`)
2. В левом меню: **Settings** → **API** → **Store API Accounts**

**Шаг 2 — Создать новый API аккаунт:**
1. Нажать кнопку **Create API Account** → выбрать **Create V2/V3 API Token**
2. Заполнить поля:
   - **Name:** `n8n-automation` (любое понятное имя)
   - **OAuth Scopes** — выставить следующие права:

| Ресурс | Уровень доступа |
|--------|----------------|
| Products | **Modify** |
| Orders | **Read-Only** |
| Webhooks | **Manage** |
| Customers | нет доступа |
| Marketing | нет доступа |

3. Нажать **Save**

**Шаг 3 — Сохранить credentials:**

Откроется окно с тремя значениями — скопировать все три **немедленно**, они показываются только один раз:
```
Client ID:     abc123xyz...
Client Secret: def456uvw...
Access Token:  ghk789rst...  ← это и есть API Token для n8n
```

> ⚠️ Access Token показывается **только один раз** — при создании. Если закрыть окно не скопировав — придётся создавать новый токен заново. Сохранить в защищённое место (1Password, Bitwarden).

**Шаг 4 — Добавить в n8n:**
1. В n8n: **Credentials** → **New** → найти **BigCommerce API**
2. Заполнить:
   - **Store Hash:** часть URL магазина (`store-XXXXX` из адреса Admin-панели)
   - **Access Token:** скопированный токен
3. Сохранить под именем `BigCommerce (icon-store)`

**Шаг 5 — Проверить подключение:**
В любом workflow с нодом BigCommerce нажать **Test Credential** — должен появиться зелёный статус Connected.

#### Что такое автоматический и ручной запуск — объяснение для оператора

BigCommerce — это сайт-магазин. У него есть **API** — программный интерфейс, через который n8n может добавлять и редактировать товары автоматически, без того чтобы оператор каждый раз заходил в административную панель и заполнял карточки вручную.

Представьте это так: вместо того чтобы открыть BigCommerce Admin → Products → Add Product → заполнить 20 полей → сохранить — n8n делает то же самое само, за секунды, для сотен товаров.

**Автоматический запуск** означает: как только оператор одобрил контент в CSV (APPROVE), n8n это видит и сам, без дополнительных действий оператора, отправляет товар в BigCommerce. Товар появляется на сайте через 1–2 минуты. Никаких дополнительных кликов не нужно.

**Ручной запуск** означает: оператор отдельно заходит в n8n и нажимает "запустить публикацию". Чуть больше контроля, но лишний шаг.

**Рекомендация для этого проекта: автоматический**, но товар появляется сначала в статусе **Draft** (черновик, не виден покупателям). Оператор проходит чекпоинт ⑤ (визуальная проверка), после чего либо вручную ставит visible, либо n8n ставит автоматически. Это даёт страховку от публикации некорректного товара.

#### Последовательность API-вызовов

n8n выполняет следующие шаги для каждого SKU:

```
1. Проверить: есть ли bc_product_id в PostgreSQL для этого SKU?
   ├── НЕТ → создать новый товар (POST)
   └── ДА  → обновить существующий (PUT)

2. POST /v3/catalog/products
   Передаёт: name, sku, description, price, weight, categories[], is_visible=false

3. POST /v3/catalog/products/{id}/images  (для каждого фото)
   Передаёт: URL изображения из S3/R2 (BigCommerce скачивает сам)
   Порядок: amazon-main первым → он становится главным фото

4. PUT /v3/catalog/products/{id}/variants  (только для размерных SKU)
   Для IC-000001-S, IC-000001-M, IC-000001-L:
   создаёт 3 варианта с разными ценами, размерами, остатками

5. PUT /v3/catalog/products/{id}/metafields  (6 namespaces)
   Сохраняет: icon_id, amazon_status, content_status, master_uri и т.д.

6. PostgreSQL: записать bc_product_id, bc_status='published', last_synced_at
```

#### Как назначаются категории

В BigCommerce категории — это папки (By Saint, By Purpose и т.д.). Каждой категории присвоен числовой ID в системе. При создании товара n8n передаёт массив ID категорий.

Маппинг происходит через таблицу соответствий: поле `saint_or_subject` → ID категории.

Примерная таблица (заполняется при настройке один раз):

#### Как перевести товар из Draft в Visible

После публикации через n8n товар находится в статусе `is_visible = false` (Draft). Покупатели его не видят. Это намеренная защита от публикации непроверенного контента.

**Вариант 1 — Вручную через BC Admin (рекомендуется для MVP):**

Единичный товар:
1. Зайти в BC Admin → Products → Product Catalog
2. Найти товар по SKU или названию
3. Открыть карточку → в блоке справа найти раздел «Storefront»
4. Переключить «Visible» в положение On
5. Нажать Save

Несколько товаров сразу (bulk):
1. Products → Product Catalog → выбрать нужные товары через чекбоксы
2. Сверху: «Bulk Actions» → «Set as Visible»
3. Подтвердить

**Вариант 2 — n8n выставляет visible автоматически:**

В workflow `bc-publish` последний шаг — PUT запрос с `is_visible = true`. Если оператор уверен что все опубликованные товары уже проверены (прошли чекпоинт ⑤), можно включить это поведение в настройках workflow.

> **Когда делать visible:** только после прохождения чекпоинта ⑤ — визуальная проверка в BC Admin: название и описание корректны, фото загружены и главное первым, цена верная, категории назначены.

#### Поле weight в мастер-базе

BigCommerce требует вес товара для расчёта стоимости доставки — без него карточка не сохранится. Поле `weight_g` уже есть в мастер-базе как необязательное, но для публикации в BC оно **должно быть заполнено**.

**Рекомендация владельцу** при заполнении мастер-базы: указывать вес в граммах в колонке `weight_g`. n8n автоматически конвертирует в фунты для BC.

Если вес конкретной иконы неизвестен — использовать типовые значения:

| Тип иконы | Примерный вес |
|-----------|:---:|
| Плоская доска, малая (до 150×200 мм) | 200–350 г |
| Плоская доска, средняя (до 200×250 мм) | 350–550 г |
| Плоская доска, большая (от 250 мм) | 550–950 г |
| Металлическая риза | 400–750 г |
| Путевая / складень | 100–250 г |

Если `weight_g` пустое — n8n использует значение по умолчанию `300` г и добавляет флаг `REVIEW_WEIGHT_DEFAULT` в отчёт. Оператор видит это и может уточнить у владельца.

#### Лимиты и обработка ошибок

**Rate limit BigCommerce Standard:** 150 запросов за 30 секунд. n8n автоматически замедляется и читает заголовок `Retry-After` в ответе API — никаких действий оператора не нужно.

| Ошибка BC API | Что делает n8n | Действие оператора |
|---------------|---------------|-------------------|
| 429 Rate Limit | Ждёт указанное время, повторяет | Ничего |
| 404 Product not found при PATCH | Переключается на POST (создать) | Ничего |
| 422 Validation error | Логирует, пропускает SKU | Проверить данные в PostgreSQL для этого SKU |
| 500 Server error | Retry 3 раза, затем FAILED | Повторить батч позже |
| Ошибка загрузки фото | Retry фото, товар создан без него | Проверить URL в `derivatives_json`, перезапустить `media-refresh` |

### Пайплайн E — Синдикация Amazon

> **Решение зафиксировано 2026-03-23:** Amazon-синдикация идёт напрямую: PostgreSQL → n8n → SP-API. BC-коннектор не используется.

#### Предварительная настройка: SP-API credentials в n8n

n8n обращается к Amazon SP-API через авторизацию LWA (Login with Amazon). Настройка выполняется один раз при деплое. Без неё workflow `amazon-syndicate` не запустится.

**Что нужно получить в Amazon Seller Central:**

Шаг 1 — Найти Seller ID и Marketplace ID:
1. Seller Central → Account → Business Information (или Account Settings)
2. Seller ID — строка вида `A2XXXXXXXXXXXXX` (видна в разных разделах SC)
3. Marketplace ID для US: **ATVPDKIKX0DER** (стандартный, не нужно искать)

Шаг 2 — Создать LWA Application (Selling Partner API app):
1. Seller Central → Apps & Services → Develop Apps
2. Нажать **Add new app client** → заполнить:
   - App name: `icon-store-n8n`
   - Consent page description: `Internal automation`
   - Login with Amazon (LWA): оставить поля как есть
3. После создания — сохранить **Client ID** и **Client Secret** (показываются при создании)

Шаг 3 — Получить Refresh Token:
1. В SC → Apps & Services → Manage Apps → найти созданное приложение
2. **Authorize** → следовать инструкции → получить `Refresh Token`
3. Refresh Token — длинная строка начинающаяся с `Atzr|...`

> ℹ️ Это упрощённый путь для self-authorized applications (продавец и разработчик — одно лицо). Полная инструкция: https://developer-docs.amazon.com/sp-api/docs/self-authorization

**Добавить в n8n:**
1. Credentials → New → найти **Amazon SP-API** (или HTTP Header Auth если нет нативного)
2. Заполнить:
   ```
   Seller ID:      A2XXXXXXXXXXXXX
   Marketplace ID: ATVPDKIKX0DER
   Client ID:      amzn1.application-oa2-client.XXXXX
   Client Secret:  amzn1.oa2-cs.v1.XXXXX
   Refresh Token:  Atzr|...
   AWS Region:     us-east-1
   ```
3. Сохранить под именем `Amazon SP-API (icon-store)`

> ⚠️ Refresh Token не истекает сам по себе, но может быть отозван при изменении прав приложения в SC. При ошибке `403 Forbidden` — перевыпустить Refresh Token через процедуру авторизации выше.

Для FBM (продавец отгружает сам) Amazon требует, чтобы каждый листинг был привязан к Shipping Template — набору правил стоимости и сроков доставки.

**Создание Shipping Template в Seller Central:**

Шаг 1:
1. Seller Central → Settings → Shipping Settings → Create New Shipping Template
2. Имя: `icon-store-standard`

Шаг 2 — Заполнить регионы и сроки:

| Регион | Тип доставки | Стоимость | Срок |
|--------|-------------|-----------|------|
| US (continental) | Standard | $0 (включена в цену) или $4.99 | 5–8 business days |
| US (continental) | Expedited | $9.99 | 2–3 business days |
| Alaska / Hawaii | Standard | $7.99 | 7–12 business days |

> Точные значения — на усмотрение владельца. Бесплатная доставка (Free Shipping) увеличивает конверсию.

Шаг 3 — Сохранить. Скопировать **Template ID** (или имя — в зависимости от SC версии).

**Привязка в n8n:**

В workflow `amazon-syndicate` в конфигурации добавить:
```

```

n8n передаёт это значение в каждый листинг через атрибут `merchant_shipping_group_name` в теле SP-API запроса.

#### Запуск — всегда ручной approve

Публикация на Amazon **всегда требует явного подтверждения оператора**. Это жёсткое правило архитектуры (принцип №6). Листинг не появится на Amazon автоматически — даже если контент approved и сайт опубликован.

Процесс: n8n формирует список SKU готовых к Amazon → уведомляет оператора → оператор проверяет и нажимает подтверждение → n8n публикует.

#### Как передаются изображения на Amazon

Оператор не загружает фото на Amazon вручную. Процесс полностью автоматический:

1. В Pipeline B фото уже обработаны и загружены в S3/R2 с публичными URL-адресами
2. При создании листинга через SP-API n8n передаёт **только URL** этих фото — например: `https://r2.example.com/icons/derived/IC-000001/amazon-main.jpg`
3. Amazon сам скачивает фото по этому URL в процессе обработки листинга
4. Порядок передачи: amazon-main первым (станет главным фото), затем amazon-gallery

Оператору ничего загружать не нужно — если фото правильно обработаны в Pipeline B и URL записаны в PostgreSQL, они автоматически попадают в листинг.

#### Полный поток

```
Оператор проверяет список готовых SKU и подтверждает публикацию
    ↓
n8n читает из PostgreSQL для каждого SKU:
    amazon_title, bullets 1–5, description, backend_search_terms,
    browse_node, item_type_keyword, amazon_price, derivatives{} URLs
    ↓
n8n → SP-API Listings API:
    PUT /listings/2021-08-01/items/{sellerId}/{merchant_sku}
    Body: { productType, attributes: { title, bullet_point[], ... },
            fulfillment_availability: [{ quantity: amazon_reserved }],
            offers: [{ price: amazon_price }] }
    GTIN не передаётся — exemption уже одобрен на аккаунте
    ↓
Amazon принимает листинг → статус: PENDING
    ↓
n8n поллит статус каждые 30 мин:
    GET /listings/2021-08-01/items/{sellerId}/{sku}
    ├── status = BUYABLE → записать ASIN, amazon_status = live ✅
    ├── status = INACTIVE/SUPPRESSED → alert оператору ⚠️
    └── всё ещё PENDING после 24 ч → alert оператору для проверки в SC
```

#### Что такое PENDING и сколько ждать

После отправки через API Amazon не публикует листинг мгновенно. Листинг проходит внутреннюю проверку:
- Изображения скачиваются и проверяются (размер, фон, запрещённые элементы)
- Категория и атрибуты валидируются
- Цена проверяется на соответствие политикам

**Типичное время:** 15 минут — 4 часа. Обычно в пределах 1 часа.
**В периоды высокой нагрузки** (крупные распродажи, праздники): до 24 часов.

n8n автоматически проверяет статус. Оператор не должен следить вручную — получит уведомление как только статус изменится.

**Причины SUPPRESSED после обработки:**

| Причина | Признак в SC | Что делать |
|---------|-------------|-----------|
| Главное фото отклонено | Image policy violation | Перепроверить фото: белый фон, нет текста, нет рамок |
| Неверный browse node | Category mismatch | Изменить `amazon_browse_node` → повторить |
| Цена ниже порога | Pricing alert | Повысить `amazon_price` в PostgreSQL → повторить |
| Отсутствует обязательный атрибут | Missing required | Добавить поле в схему → заполнить для SKU → повторить |

**FBM: управление заказами:**
```
n8n poll каждые 30 мин → Orders API GET /orders?OrderStatuses=Unshipped
    ↓
Новые заказы → записать в PostgreSQL (orders) → уведомить оператора
    ↓
Оператор отгружает со склада
    ↓
Вводит трекинг-номер в n8n-форму (или напрямую в SC)
    ↓
n8n → Orders API: подтвердить отгрузку с tracking
    → Amazon помечает заказ как отправленный, покупатель получает email
    ↓
PostgreSQL: обновить статус заказа, уменьшить amazon_reserved
```

#### Как оператор физически подтверждает публикацию на Amazon

Amazon-синдикация никогда не запускается автоматически — каждый раз требуется явное действие оператора. Три варианта реализации:

**Вариант 1 — n8n Form (рекомендуется для MVP)**

n8n Form Trigger создаёт простую веб-страницу по постоянной ссылке. Оператор видит список готовых SKU и нажимает «Подтвердить».

Процесс:
1. n8n находит SKU с `content_status = 'approved'` и `amazon_status = 'not_listed'`
2. Отправляет оператору уведомление: «Готовы к Amazon: 5 SKU. Проверить: [ссылка]»
3. Оператор переходит по ссылке → видит таблицу: SKU, название, цена, browse_node
4. Проверяет данные, нажимает «Опубликовать» → n8n запускает `amazon-syndicate`

Ссылку нужно сохранить как закладку. Формат: `https://your-n8n.example.com/form/amazon-approve`

**Вариант 2 — Кнопка Run в n8n Dashboard**

Оператор заходит в n8n напрямую → открывает workflow `amazon-syndicate` → нажимает «Run». Перед запуском n8n показывает список SKU которые будут обработаны. Простой и надёжный вариант для старта.

**Вариант 3 — Email с кнопками Approve/Reject**

n8n отправляет email со списком SKU и двумя кнопками: «Подтвердить» / «Отложить». Нажатие кнопки активирует webhook в n8n через уникальный токен в URL. Удобно если оператор работает с почтой а не с n8n Dashboard.

> Рекомендация: начать с **Варианта 2** (нет доп. настройки), перейти на **Вариант 1** когда поток регулярный.

#### productType для Amazon SP-API — откуда берётся

`productType` — обязательное поле SP-API. Оно указывает Amazon в какую схему атрибутов укладывать товар. Разные категории товаров имеют разные обязательные поля.

**Для этого каталога — hardcoded значение в n8n**

Все товары каталога — иконы, то есть одна категория. `productType` задаётся один раз в конфигурации workflow `amazon-syndicate` и не меняется для каждого SKU отдельно.

**Кандидаты для иконного магазина:**

| productType | Описание | Подходит |
|-------------|----------|:---:|
| `HOME_DECOR` | Предметы декора для дома | ✅ Основной кандидат |
| `WALL_ART` | Настенное искусство | ✅ Альтернатива |
| `POSTER` | Постеры | ⚠️ Не идеален для физического объекта |

**Как окончательно определить правильный productType:**
1. После регистрации SC: Seller Central → Add a Product → ввести «orthodox icon»
2. Amazon предложит категорию и схему атрибутов — это и есть нужный productType
3. Занести в конфигурацию workflow (один раз)

**Если в каталоге появятся разные типы товаров** (например, иконостасы отдельным productType) — добавить поле `amazon_product_type` в мастер-базу как необязательное. n8n использует его если заполнено, иначе берёт дефолт из настроек workflow.

#### browse_node — кто генерирует, кто проверяет

`browse_node` — числовой ID узла в дереве категорий Amazon (например: `3764461`). Обязательное поле для листинга.

**Источник значения — двухэтапный процесс:**

1. **Pipeline C (Claude API)** — Claude предлагает `browse_node` на основе текста карточки. Значение попадает в output JSON → сохраняется в поле `amazon_browse_node` в PostgreSQL.

2. **Оператор при review CSV** — проверяет предложенный browse_node в колонке I и при необходимости исправляет вручную. Это обязательная часть чекпоинта ④.

**Как проверить правильность browse_node:**
- SC → Add a Product → поиск «orthodox icon» → смотреть предложенный node ID
- Или найти конкурентный листинг → скопировать ASIN → SP-API: `GET /catalog/2022-04-01/items/{asin}` → в ответе поле `classifications[].classificationId`

Финальный browse_node нужно протестировать на первых 10 SKU и сравнить impressions через 2–4 недели. Один browse_node для всего каталога на старте — правильная стратегия (см. Spec-4, раздел 18).

#### ASIN — поле, когда записывается, как хранится

ASIN (Amazon Standard Identification Number) — уникальный ID товара на Amazon. Присваивается Amazon после одобрения листинга.

**Поле в PostgreSQL:** `asin VARCHAR(20)` — уже есть в DDL (Spec-1, таблица `icons`). По умолчанию `NULL`.

**Когда и как записывается:**

1. n8n отправил листинг через SP-API
2. Amazon одобрил → листинг стал `BUYABLE`
3. n8n читает ASIN через poll:
   ```
   GET /listings/2021-08-01/items/{sellerId}/{sku}
   → ответ содержит summaries[0].asin
   ```
4. n8n выполняет обновление PostgreSQL:
   ```sql
   UPDATE icons
   SET asin             = 'B0XXXXXXXXX',
       amazon_status    = 'live',
       last_synced_at   = NOW()
   WHERE merchant_sku = 'IC-000001';
   ```
5. n8n также обновляет BC metafield `catalog.source/asin` — чтобы ссылка «Buy on Amazon» могла отображаться на сайте (см. раздел ниже)

**Как оператор проверяет полученные ASIN:**
```sql
SELECT merchant_sku, asin, amazon_status, last_synced_at
FROM icons
WHERE amazon_status = 'live'
ORDER BY last_synced_at DESC;
```

#### Retry — что делать если листинг не прошёл

**Сценарий 1 — API вернул ошибку при отправке:**

| Код ошибки | Значение | Что делать |
|------------|----------|------------|
| `400 Bad Request` | Неверный формат данных | Проверить поля: title непустой, browse_node числовой, цена > 0 |
| `403 Forbidden` | Проблема с авторизацией | Проверить SP-API credentials в n8n |
| `429 Throttling` | Слишком частые запросы | n8n retry автоматически через 60 сек |
| `InvalidInput` | Поле не соответствует схеме productType | Проверить обязательные атрибуты для данного productType в SC |

Шаги retry при API ошибке:
1. n8n → Executions → найти упавший `amazon-syndicate`
2. Открыть → увидеть на каком SKU ошибка и текст ошибки
3. Исправить данные в PostgreSQL (через pgAdmin или DBeaver)
4. В n8n нажать «Retry» для этого execution — только упавшие SKU

**Сценарий 2 — Листинг стал SUPPRESSED:**

Оператор получает alert → идёт в SC → Manage Inventory → находит suppressed → читает причину:

1. Ошибка в данных — исправить в PostgreSQL:
   ```sql
   -- Пример: неверный browse_node
   UPDATE icons SET amazon_browse_node = '3764461'
   WHERE merchant_sku = 'IC-000001';
   ```
2. Ошибка в фото — переделать в Pipeline B → загрузить новый master в S3/R2 → обновить URL в `derivatives_json`
3. После исправления — запустить `amazon-syndicate` снова для этого SKU (старый листинг обновится через PUT)

#### Ссылка на Amazon с сайта BigCommerce

После получения ASIN, на странице товара в BigCommerce можно добавить кнопку «View on Amazon». Это выгодно: часть покупателей доверяет Amazon больше незнакомого сайта, и покупка всё равно происходит (просто через другой канал).

**Amazon URL строится по шаблону:** `https://www.amazon.com/dp/{ASIN}`

n8n после получения ASIN автоматически формирует URL и записывает его в BC metafield `catalog.source/amazon_url`. Оператор ничего не делает вручную.

**Три варианта отображения на сайте:**

| Вариант | Сложность | Результат |
|---------|-----------|-----------|
| **Script Manager (рекомендуется)** | Минимальный JS, один раз | Кнопка «View on Amazon» автоматически на всех PDP где есть ASIN |
| Custom Field в теме | Правка шаблона темы | Кнопка в нужном блоке страницы товара |
| HTML в описании товара | Нет кода | Ссылка внутри текста описания |

**Script Manager — как добавить (один раз):**
1. BC Admin → Storefront → Script Manager → Create a Script
2. Name: «Amazon Buy Button», Location: Footer, Pages: Product Pages
3. Код:
```javascript
(function() {
  // Читает amazon_url из метаданных страницы
  var meta = document.querySelector('meta[name="amazon-url"]');
  if (!meta || !meta.content) return;
  var btn = document.createElement('a');
  btn.href = meta.content;
  btn.target = '_blank';
  btn.rel = 'noopener';
  btn.className = 'button button--secondary';
  btn.style.marginTop = '10px';
  btn.textContent = 'View on Amazon';
  var addToCart = document.querySelector('[data-button-type="add-cart"]');
  if (addToCart) addToCart.parentNode.insertBefore(btn, addToCart.nextSibling);
})();
```

n8n при обновлении BC metafield записывает amazon_url, тема рендерит meta-тег, скрипт подхватывает и отображает кнопку.

> **Решение о показе кнопки — за владельцем.** Кнопка «View on Amazon» направляет трафик на Amazon, что ослабляет прямые продажи через сайт. Технически включается/выключается одним параметром в настройках workflow или Script Manager.

### Пайплайн F — Инвентаризация

> **FBM, единый физический склад, два логических резерва** (зафиксировано 2026-03-23)

#### Как работает BC Webhook — объяснение

Webhook — это механизм оповещения. Когда в BigCommerce происходит событие (например, продажа), BigCommerce сам отправляет уведомление на адрес n8n: "только что продали 1 единицу товара IC-000001". n8n получает это мгновенно и обновляет PostgreSQL.

Это как push-уведомление на телефон — не надо каждые 5 минут проверять "а не продалось ли что-нибудь", сообщение приходит само.

**Используемые события:**
- `store/order/statusUpdated` — заказ создан/оплачен → уменьшить `site_stock`
- `store/product/inventory/updated` — ручное изменение остатка в BC Admin → синхронизировать с PostgreSQL

#### Как технически обновляется amazon_reserved через SP-API

Когда `amazon_reserved` меняется в PostgreSQL (продажа или пополнение), n8n сообщает об этом Amazon через SP-API:

```
SP-API: PATCH /listings/2021-08-01/items/{sellerId}/{sku}
Body: {
  "fulfillment_availability": [{
    "fulfillmentChannelCode": "DEFAULT",
    "quantity": <новое значение amazon_reserved>
  }]
}
```

Amazon обновляет отображаемое количество в наличии на листинге. Если `amazon_reserved = 0`, листинг показывает "Currently unavailable" и не принимает заказы — это безопасно.

#### Как оператор вводит пополнение склада

В n8n создаётся простая форма (n8n Form Trigger). Оператор открывает ссылку → заполняет:
- SKU товара (или выбирает из списка)
- Количество поступившего товара
- Куда распределить: только на сайт / только Amazon / поровну

n8n пересчитывает резервы и обновляет оба канала. Никакого SQL не нужно.

**Полный поток изменения остатков:**

```
Продажа на сайте:
BC принимает заказ → webhook → n8n → site_stock - 1 в PostgreSQL
    → если site_stock стал 0: PATCH BC → товар скрывается автоматически
    → если site_stock < 3: alert оператору

Продажа на Amazon (FBM):
Orders API poll → новый заказ → amazon_reserved - 1 в PostgreSQL
    → уведомление оператору об отгрузке
    → если amazon_reserved < 3: alert оператору

Пополнение склада (оператор):
n8n форма → оператор вводит кол-во и распределение
    → PostgreSQL обновлён
    → PATCH BC inventory (site_stock)
    → PATCH SP-API listing quantity (amazon_reserved)
```

**Алерты:**
- `site_stock` < 3 → предупреждение
- `amazon_reserved` < 3 → предупреждение
- Оба канала активны, `site_stock + amazon_reserved` < 2 → срочный алерт
- `site_stock` = 0 при активном BC → автоскрытие товара на сайте

#### Первоначальная регистрация BC Webhook — одноразовая настройка

BC webhook регистрируется один раз при первом деплое. Без этого n8n не будет получать уведомления о продажах.

**Что такое регистрация webhook:** BigCommerce должен «знать» куда отправлять уведомления. Регистрация — это один API-запрос к BC который говорит: «При событии X отправляй уведомление на URL Y».

**Пошаговая инструкция:**

**Шаг 1 — Получить n8n webhook URL:**
1. В n8n создать нод «Webhook» (trigger) в workflow `inventory-sync`
2. Скопировать **Production URL** — выглядит как: `https://your-n8n.example.com/webhook/bc-orders`

**Шаг 2 — Найти данные для BC API:**
- `{STORE_HASH}` — BC Admin → Advanced Settings → API Accounts → Store Hash (короткий буквенно-цифровой код)
- `{BC_API_TOKEN}` — BC Admin → Advanced Settings → API Accounts → «Create API Account» → дать права: Orders (read-only), Products (read & write) → скопировать токен

**Шаг 3 — Зарегистрировать первый webhook (продажи):**

Выполнить через нод «HTTP Request» в n8n (один раз) или через Postman:
```
POST https://api.bigcommerce.com/stores/{STORE_HASH}/v2/hooks
Headers:
  X-Auth-Token: {BC_API_TOKEN}
  Content-Type: application/json

Body:
{
  "scope": "store/order/statusUpdated",
  "destination": "https://your-n8n.example.com/webhook/bc-orders",
  "is_active": true
}
```

**Шаг 4 — Зарегистрировать второй webhook (изменения инвентаря вручную):**
```json
{
  "scope": "store/product/inventory/updated",
  "destination": "https://your-n8n.example.com/webhook/bc-inventory",
  "is_active": true
}
```

**Шаг 5 — Проверить что оба webhook активны:**
```
GET https://api.bigcommerce.com/stores/{STORE_HASH}/v2/hooks
```
В ответе должны быть два объекта с `"is_active": true`.

> Webhook остаётся активным пока n8n работает и доступен по указанному URL. Если URL изменится (например, при смене домена VPS) — зарегистрировать заново.

#### Что происходит если n8n временно недоступен — retry BigCommerce

BigCommerce не теряет события если n8n в момент продажи недоступен (перезагрузка VPS, обновление n8n, кратковременный сетевой сбой).

**Механизм повтора BC:**

BigCommerce повторно пробует отправить webhook-уведомление **до 24 часов** после первой неудачи. Интервалы повтора увеличиваются постепенно:

| Попытка | Через сколько после предыдущей |
|---------|-------------------------------|
| 1-я (исходная) | Сразу при событии |
| 2-я | ~1 минута |
| 3-я | ~5 минут |
| 4-я | ~30 минут |
| Далее | Каждые несколько часов до 24 ч |

**Что это означает на практике:**
- Если VPS перезагружается и n8n недоступен 10–15 минут — BC автоматически повторит и событие не потеряется
- После восстановления n8n получит все накопившиеся события и обновит PostgreSQL
- Оператор не должен ничего делать вручную при кратковременных сбоях

**Граничный случай: n8n недоступен более 24 часов**

Если VPS был недоступен дольше суток (сбой хостинга, неоплаченный сервер) — BC перестаёт повторять. В этом случае:
1. После восстановления n8n — запустить вручную сверку `inventory-sync` workflow
2. Проверить расхождение через ежедневный автоотчёт (он покажет разницу)
3. При существенном расхождении — пройти процедуру «Что делать при расхождении BC ↔ PostgreSQL» ниже

> **Рекомендация:** настроить внешний мониторинг VPS (UptimeRobot, Hetrix Tools — бесплатные планы), чтобы оператор получал SMS/email при простое более 5 минут. Это предупредит потерю событий до истечения 24-часового окна BC.

#### Возвраты и отмены заказов

**Отмена на Amazon (до отгрузки):**
- Amazon → n8n получает статус `Canceled` через Orders API poll
- n8n автоматически: `amazon_reserved + 1` (единица возвращается в резерв)
- Оператор получает уведомление: «Заказ #{id} отменён, инвентарь восстановлен»

**Возврат на Amazon (после доставки):**
- Покупатель открыл возврат → Amazon создаёт Return
- n8n видит Return со статусом `ReturnReceived` → `amazon_reserved + 1`
- Оператор получает уведомление и решает что делать физически: вернуть товар в сток или списать
- Если товар негоден (не возвращать в сток), скорректировать вручную:
  ```sql
  -- Выполнить ТОЛЬКО если автовосстановление уже произошло и товар на самом деле не возвращается
  UPDATE icons
  SET amazon_reserved = amazon_reserved - 1
  WHERE merchant_sku = 'IC-000001';
  ```

**Отмена на BigCommerce:**
- BC webhook `store/order/statusUpdated` → статус `Cancelled` → n8n: `site_stock + 1`
- Если товар уже упаковали — оператор физически возвращает его на склад, цифра обновилась автоматически

**Возврат на BigCommerce:**
- BC не имеет нативного API-события для возвратов (в отличие от Amazon)
- Ручной процесс: оператор получил физически возвращённый товар → открывает форму n8n пополнения → вводит +1 к `site_stock` с пометкой «return»

**Общее правило при любом сомнении в цифрах:** посчитать реальный склад → сравнить с `site_stock + amazon_reserved` из PostgreSQL → через форму пополнения привести к правильному значению → запись в `audit_log`.

#### Как оператор открывает форму пополнения склада

n8n Form Trigger создаёт постоянный URL — он не меняется пока workflow активен.

**Как найти URL:**
1. В n8n → открыть workflow `inventory-replenishment`
2. Кликнуть на нод «Form Trigger»
3. Скопировать **Production URL** — выглядит как: `https://your-n8n.example.com/form/inventory`

**Рекомендации по хранению ссылки:**
- Добавить в браузер как закладку под именем «📦 Пополнение склада»
- Добавить в Telegram-бот или email-подпись для быстрого доступа
- При наличии физического склада — распечатать QR-код и повесить в месте приёмки

**Поля формы:**
1. SKU товара (выпадающий список из PostgreSQL — только активные SKU)
2. Количество поступившего товара (число)
3. Распределение: «Только сайт» / «Только Amazon» / «50/50» / «Вручную задать»
4. Примечание (опционально): «Возврат», «Новая партия», и т.д.

n8n пересчитывает резервы, обновляет PostgreSQL, синхронизирует BC и Amazon. Никакого SQL не нужно.

#### Что делать при расхождении BC ↔ PostgreSQL

**Как обнаруживается расхождение — ежедневная автосверка:**

n8n по расписанию (раз в сутки) выполняет сверку:
```
n8n → BC API: GET /v3/catalog/products/{id}/variants → читает inventory_level
n8n → PostgreSQL: читает site_stock
Разница > 0 → пишет в лог, уведомляет оператора
```

Оператор получает уведомление: `«Расхождение: IC-000001 — BC=5, PostgreSQL=3»`

**Шаг 1 — Физически проверить реальный склад:** посчитать сколько единиц IC-000001 есть на руках.

**Шаг 2 — Найти причину расхождения:**

| Причина | Как определить | Источник истины |
|---------|---------------|-----------------|
| Оператор вручную изменил остаток в BC Admin | BC Admin → Products → история изменений | PostgreSQL |
| BC webhook не сработал при продаже | n8n → Executions — нет выполнения за нужное время | Реальный склад |
| Товар добавлен в BC напрямую (не через n8n) | BC Admin — есть товар без `icon_id` в metafield | PostgreSQL |
| Реальный склад совпадает с BC, а PostgreSQL отстал | Физический счёт = BC | Обновить PostgreSQL |

**Шаг 3 — Привести к правильному значению через форму пополнения:**
- Форма принимает абсолютное число: «сейчас на складе X единиц»
- n8n обновляет PostgreSQL → синхронизирует BC → синхронизирует Amazon

**Шаг 4 — Зафиксировать в audit_log:**
```sql
INSERT INTO audit_log (icon_id, action, old_value, new_value, performed_by, notes)
SELECT icon_id, 'INVENTORY_RECONCILE',
       site_stock::TEXT, '5',
       'operator', 'Manual reconcile: BC=5, PG was 3, real stock=5'
FROM icons WHERE merchant_sku = 'IC-000001';
```

Если расхождения повторяются регулярно — проверить что BC webhook активен (раздел выше) и не истёк ли срок токена API.

### Пайплайн G — Batch Engine (сквозной)

#### Что такое batch engine и зачем он нужен

Batch engine — это общий механизм управления пакетными операциями. Представьте: нужно опубликовать 200 товаров в BigCommerce. Если что-то сломается на 150-м, без batch engine придётся начинать заново все 200. С batch engine — система знает что 149 уже сделано, продолжит с 150-го.

#### Жизненный цикл батча

```
CREATED          — батч создан, задания перечислены
    ↓
VALIDATING       — проверка входных данных
    ↓
VALIDATED        — данные корректны, можно запускать
    ↓
RUNNING          — обработка идёт
    ↓
    ├── COMPLETE                  — всё успешно ✅
    ├── PARTIALLY_COMPLETE        — часть успешно, часть с ошибками ⚠️
    ├── FAILED_WITH_EXCEPTIONS    — критические ошибки ❌
    └── ROLLED_BACK               — откат выполнен (после критического сбоя)
```

#### Идемпотентность — защита от двойной обработки

Каждый item в батче хранит:
- `source_hash` — хэш входных данных (что мы хотели сделать)
- `target_hash` — хэш результата в целевой системе (что там уже есть)

Если они совпадают — товар уже обработан корректно, система пропускает его. Это значит: **можно безопасно запустить батч повторно** — ничего лишнего не произойдёт.

#### Параллелизм в n8n

n8n обрабатывает items в батче **последовательно**, но быстро. Для 100 SKU публикация в BC займёт 3–5 минут (150 запросов/30 сек лимит). Это не "медленно" — оператор запустил и занялся другим делом.

Исключение: Claude Batches API обрабатывает все SKU **параллельно** на стороне Anthropic — 100 SKU занимают примерно столько же, сколько 1 SKU.

#### Retry: пороги и интервалы

| Попытка | Пауза перед следующей |
|---------|----------------------|
| 1-я (первичная) | — |
| 2-я (retry 1) | 5 минут |
| 3-я (retry 2) | 15 минут |
| 4-я (retry 3) | 30 минут |
| Далее | Статус FAILED, ждёт оператора |

Итого: n8n сам пытается исправить ошибку в течение ~50 минут. Если не получилось — сообщает оператору.

#### Что делает оператор при PARTIALLY_COMPLETE

Это самый частый нештатный статус. Означает: большинство прошло успешно, несколько SKU упали.

**Шаги оператора:**
1. Получить уведомление: "Batch bc-publish: 47 success, 3 failed"
2. Открыть n8n → Executions → найти батч
   Или выполнить SQL:
   ```sql
   SELECT sku, error_code, error_detail
   FROM batch_items
   WHERE batch_id = '<id>' AND status = 'FAILED';
   ```
3. По таблице ошибок из Spec-5 — понять причину для каждого SKU
4. Исправить данные в PostgreSQL (если проблема в данных)
5. В n8n: нажать "Retry failed items" для этого батча
6. n8n обработает только 3 упавших SKU, успешные 47 не трогает

**Откат при критическом сбое (ROLLED_BACK):**

| Тип операции | Как откатывается |
|-------------|-----------------|
| Контент | Восстановить предыдущую версию из `content_versions` |
| Изображения | Переключить `derivatives_json` на предыдущий набор URI |
| Цены | Переприменить предыдущий `pricing_rule_id` |
| Публикация BC | Вернуть `bc_status = 'draft'`, убрать с сайта |
| **Публикация Amazon** | **Выставить `quantity = 0` через SP-API (мягкий) или DELETE listing (жёсткий); `amazon_status = 'not_listed'`; ASIN сохранить** |

**Детали отката Amazon-листинга:**

Вариант A — мягкий (рекомендуется): скрыть листинг не удаляя. Товар исчезает с продажи, ASIN сохраняется. Можно быстро вернуть:
```
SP-API: PATCH /listings/2021-08-01/items/{sellerId}/{sku}
Body: { "fulfillment_availability": [{ "fulfillmentChannelCode": "DEFAULT", "quantity": 0 }] }
```

Вариант B — жёсткий: удалить листинг. Используется только при серьёзных ошибках (неправильный productType, нарушение политик):
```
SP-API: DELETE /listings/2021-08-01/items/{sellerId}/{sku}
```

PostgreSQL при откате Amazon (оба варианта):
```sql
UPDATE icons
SET amazon_status      = 'not_listed',
    publication_notes  = 'ROLLED_BACK: ' || NOW()::TEXT
WHERE merchant_sku = 'IC-000001';
```

#### Какие workflows используют Batch Engine

Batch engine — архитектурный паттерн, реализованный в следующих workflows:

| Workflow | Что делает | Batch Engine |
|----------|-----------|:---:|
| `source-import` | Загрузка мастер-базы в PostgreSQL | ✅ |
| `source-patch` | Обновление существующих SKU | ✅ |
| `content-generate` | Генерация контента через Claude API | ✅ |
| `content-review-apply` | Применение решений APPROVE/REJECT/REGEN из CSV | ✅ |
| `bc-publish` | Публикация в BigCommerce | ✅ |
| `amazon-syndicate` | Публикация на Amazon через SP-API | ✅ |
| `price-update` | Пересчёт и обновление цен по правилам | ✅ |
| `media-refresh` | Перегенерация производных фото (при замене мастера) | ✅ |
| `image-process` | Обработка фото: Stage B → S3/R2 | ⚠️ Частично — каждый SKU независим, нет общего batch_run |
| `inventory-replenishment` | Пополнение склада через форму | ❌ Разовая операция |

**Итого: 8 из 10 workflows используют batch engine.** Для этих 8 workflows оператор может безопасно прервать и перезапустить без потери прогресса.

#### Как workflow подключается к Batch Engine в n8n

Для оператора технические детали не важны — их настраивает разработчик при деплое. Но понимать логику полезно для диагностики.

Batch engine — это не отдельный сервис, а набор нодов внутри каждого workflow плюс таблицы `batch_runs` и `batch_items` в PostgreSQL.

**Схема внутри workflow (упрощённо):**

```
[Trigger] → [Read готовых SKU из PostgreSQL]
               ↓
         [Создать запись batch_run: status = RUNNING]
               ↓
         [Loop: по каждому SKU]
               ↓
         [Проверить: source_hash == target_hash?]
            ├── Совпадает → пропустить (уже сделано)
            └── Отличается → выполнить работу
                                ↓
                          [Обновить batch_items: status = SUCCESS или FAILED]
               ↓ (после loop)
         [Подсчитать: все SUCCESS → COMPLETE, есть FAILED → PARTIALLY_COMPLETE]
               ↓
         [Обновить batch_run: финальный статус]
               ↓
         [Уведомить оператора]
```

**Что это даёт оператору на практике:**
- n8n → Executions: каждый запуск workflow = одна строка в `batch_runs`
- При ошибке: Retry в n8n → workflow прочитает `batch_items`, найдёт FAILED, обработает только их, успешные пропустит

**Все батчи за последние 7 дней:**
```sql
SELECT
  workflow_name,
  status,
  total_items,
  success_count,
  failed_count,
  started_at,
  ROUND(EXTRACT(EPOCH FROM (COALESCE(completed_at, NOW()) - started_at)) / 60, 1) AS duration_min
FROM batch_runs
WHERE started_at > NOW() - INTERVAL '7 days'
ORDER BY started_at DESC;
```

**Активный батч прямо сейчас:**

#### Максимальный размер батча

Ограничения зависят от конкретного workflow:

| Workflow | Практический лимит | Рекомендуемый размер | Причина ограничения |
|----------|:-----------------:|:--------------------:|---------------------|
| `source-import` | ~5 000 строк | 500 за раз | Память n8n при обработке JSON |
| `content-generate` | ~1 000 SKU | 20–50 за раз | Claude Batches API: оплата растёт |
| `bc-publish` | ~1 000 SKU | 100–200 за раз | BC Rate Limit 150 req/30 sec |
| `amazon-syndicate` | 50 SKU | 10–20 за раз | SP-API throttle: 5 req/sec |
| `price-update` | ~1 000 SKU | 200 за раз | BC Rate Limit |
| `media-refresh` | ~500 SKU | 50 за раз | Python worker и S3 bandwidth |

**Жёсткий лимит — оперативная память VPS:**
- VPS с 2 GB RAM: безопасный предел ~300–500 items за один запуск
- VPS с 4 GB RAM: до ~1 500 items

**Рекомендация при запуске полного каталога (1 000+ SKU):**

Разбить на батчи по 200 SKU и запускать последовательно. n8n не потеряет прогресс между батчами — каждый завершённый item зафиксирован в `batch_items` с `status = SUCCESS`. При следующем запуске система автоматически пропустит уже обработанные.

## 8. Управление ценами

Цена не живёт "просто в BigCommerce". Она рассчитывается:

```
base_cost (в master card)
    × markup_multiplier (в таблице pricing_rules)
    + handling_fee
    = site_price

amazon_price = site_price × amazon_fee_factor
    (amazon_fee_factor: 1.0 = паритет, 1.15 = покрытие referral fee 15%)
```

**Пример правила "Standard Markup":**
```
markup_multiplier : 2.5
handling_fee      : $0
amazon_fee_factor : 1.0   (паритет с сайтом)
min_site_price    : $15   (floor — ниже не публиковать)
```

При изменении правила → batch `price-update` → обновляет все каналы пакетно.
Ручная правка цены в BigCommerce — только исключение, фиксируется в `audit_log`.

### 16.3 Python worker — контракт функций

```python
# image_worker.py — контракт (не production-ready)

class ImageWorker:

    def stage_a_to_b(self, sku: str, source_path: str) -> StageResult:
        """
        Incoming → Approved Master.
        Шаги:
          1. Загрузить файл из source_path (Drive path или URL)
          2. validate_input(): размер ≥ 2000px по длинной стороне, RGB/RGBA
          3. remove_background(): Remove.bg API → rembg fallback
          4. normalize_image(): яркость 240–255 (среднее фона), резкость, WB
          5. add_white_background(): composite на 2000×2000 белый canvas с padding 5%
          6. upload_to_storage(): icons/originals/{sku}/master.jpg
          7. db_update(): approved_master_uri, image_qa_status='approved'
        Returns: StageResult(success, uri, qa_notes)
        """

    def stage_b_to_c(self, sku: str, master_uri: str) -> DerivativesResult:
        """
        Approved Master → Published Derivatives.
        Шаги:
          1. Загрузить master из S3/R2
          2. Для каждого варианта из DERIVATIVE_SPECS:
             - resize_and_pad() или resize_and_crop() в зависимости от типа
             - encode() в нужный формат с заданным quality
             - upload_to_storage(): icons/derived/{sku}/{variant}.ext
          3. db_update(): derivatives JSONB с URL всех вариантов
        Returns: DerivativesResult(success, derivatives_map, errors)
        """

    def validate_input(self, img) -> ValidationResult:
        """
        Проверки:
          - min_dimension >= 2000px
          - mode in ('RGB', 'RGBA', 'L')  — конвертировать если нужно
          - file_size < 50MB
          - not_corrupt: можно открыть без ошибок
        """

    def remove_background(self, img) -> Image:
        """
        1. Попытка: Remove.bg API (платная, качество выше)
        2. Fallback: rembg (локальная модель u2net)
        3. Если оба упали: image_qa_status='processing', алерт оператору
        """

    def normalize_image(self, img) -> Image:
        """
        - Auto-levels: растянуть гистограмму если среднее < 200
        - Sharpen: unsharp mask (radius=1, percent=50, threshold=3)
        - Color balance: нейтрализовать цветовой сдвиг фона (< 0.05 delta)
        """
```

### 17.6 Валидация output contract

```python
AMAZON_FIELD_LIMITS = {
    "title":                200,   # chars
    "bullet_1":             500,   # chars (Amazon actual max, keep <300 for UX)
    "bullet_2":             500,
    "bullet_3":             500,
    "bullet_4":             500,
    "bullet_5":             500,
    "description_html":     2000,  # chars
    "backend_search_terms": 249,   # bytes (не chars!)
}

def validate_content_output(obj: dict) -> list[str]:
    flags = []
    amazon = obj.get("amazon", {})
    for field, limit in AMAZON_FIELD_LIMITS.items():
        value = amazon.get(field, "")
        if field == "backend_search_terms":
            if len(value.encode("utf-8")) > limit:
                flags.append(f"REVIEW_LENGTH_VIOLATION:{field}")
        elif len(value) > limit:
            flags.append(f"REVIEW_LENGTH_VIOLATION:{field}")
    # Проверка обязательных полей
    for required in ["title", "bullet_1", "browse_node"]:
        if not amazon.get(required):
            flags.append(f"REVIEW_MISSING_DATA:{required}")
    return flags
```

| Backend keywords | ≤ 249 bytes | `amazon_backend_search_terms` |
| Browse node | Числовой ID | `amazon_browse_node` |
| Item type keyword | Строка | `amazon_item_type_keyword` |
| Condition | New | Хардкод |
| Price | В долларах USD | `amazon_price` из PostgreSQL |
| Fulfillment | MFN (FBM — продавец отгружает сам) | По умолчанию для FBM |

### 18.3 Путь синдикации — финальное решение

> BC-коннектор для Amazon **не используется**. Нативный BC Amazon Channel не поддерживает FBM-специфичные поля листинга. Путь: **n8n → SP-API Listings API напрямую** (зафиксировано 2026-03-23, см. Pipeline E).

# Spec-5: Operator Playbook

## 19. Процедура оператора — от нового SKU до публикации в оба канала

> Этот раздел описывает не потоки данных (они в разделе 7), а **что делает человек** на каждом шаге, через какой интерфейс, и как он знает, что всё прошло корректно.

---

### 19.1 Матрица статусов — полная карта разблокировки

Каждый SKU проходит через последовательность статусов. Шаг не начинается, пока предыдущий не завершён.

```
[НОВЫЙ SKU]
    ↓  Пайплайн A (оператор загружает CSV)
image_qa_status = 'incoming'
content_status  = 'not_started'
bc_status       = 'draft'
amazon_status   = 'not_listed'

    ↓  Пайплайн B Stage A→B (авто + ручной QA)
image_qa_status = 'approved'          ← разблокирует генерацию контента

    ↓  Пайплайн C (авто + оператор ставит APPROVE)
content_status  = 'approved'          ← разблокирует публикацию в оба канала

    ↓  Пайплайн D (авто)
bc_status       = 'published'         ← товар виден на сайте

    ↓  Пайплайн E (авто после финального approve)
amazon_status   = 'live'              ← листинг активен на Amazon
```

**Правило блокировки:**

| Нужно для BigCommerce | Нужно для Amazon |
|-----------------------|-----------------|
| `image_qa_status = approved` | `image_qa_status = approved` |
| `content_status = approved` | `content_status = approved` |
| — | `bc_status = published` |
| — | GTIN Exemption одобрен |

---

### 19.2 Разовая настройка системы (один раз перед первым SKU)

```
□ 1. Развернуть VPS: Docker Compose с n8n + PostgreSQL
□ 2. Применить DDL из Spec-1 (раздел 15)
□ 3. Создать S3/R2 бакет, структуру папок (icons/incoming/, originals/, derived/)
□ 4. Настроить credentials в n8n:
      - BigCommerce API key (Catalog r/w, Webhook w)
      - Amazon SP-API (Listings w, Orders r/w)
      - S3/R2 (upload, read, list)
      - Claude API (Messages + Batches)
      - Remove.bg API key
□ 5. Подключить Google Drive к n8n (webhook на новые файлы в incoming/)
□ 6. Получить GTIN Exemption в Amazon Seller Central
□ 7. Провести тест с 1 тестовым SKU end-to-end (не публиковать, только проверить пайплайн)
□ 8. Настроить email/Slack уведомления в n8n для алертов
```

---

### 19.3 Шаг 1 — Загрузка каталога (Пайплайн A)

**Триггер:** получен файл от поставщика (CSV или Excel).

**Действия оператора:**
1. Проверить структуру файла: обязательные колонки — `title_raw`, `saint_or_subject`, `material`, `size_w_mm`, `size_h_mm`.
2. Открыть n8n → запустить workflow `source-import`, указать путь к файлу.
3. Дождаться завершения — n8n выводит сводку: `N created / M skipped (dup SKU) / K errors`.
4. При наличии ошибок — открыть `batch_items` (статус ERROR), исправить данные, перезапустить для ошибочных строк.

**Как знать, что всё ок:**
```sql
SELECT merchant_sku, image_qa_status, content_status
FROM icons
WHERE created_at > NOW() - INTERVAL '1 hour'
ORDER BY merchant_sku;
-- Все новые SKU должны иметь image_qa_status='incoming', content_status='not_started'
```

---

### 19.4 Шаг 2 — Загрузка изображений (Пайплайн B)

**Действия оператора:**
1. Создать папку в Google Drive: `incoming/{IC-000001}/` (имя папки = merchant_sku).
2. Загрузить исходные фото в эту папку (raw-001.jpg, raw-002.jpg …).
3. n8n-webhook срабатывает автоматически, запускает `image_worker.stage_a_to_b()`.
4. При успехе: n8n уведомляет оператора, статус `image_qa_status = approved`, производные генерируются автоматически.
5. При ошибке: n8n присылает алерт с кодом ошибки (см. таблицу QA в Spec-2).

**Разбор ошибок:**

| Код ошибки | Что делать |
|------------|-----------|
| `IMG_TOO_SMALL` | Запросить у поставщика пересъёмку ≥ 2000px |
| `IMG_BG_REMOVAL_FAILED` | Отправить фото ретушёру, вернуть в approved-папку |
| `IMG_NO_SUBJECT` | Проверить кадрирование, переснять |
| `IMG_COLOR_CAST` | n8n предупреждает, но не блокирует — оператор решает approve/reject вручную через SQL |

**Ручной approve при нестандартных случаях:**
```sql
UPDATE icons
SET image_qa_status = 'approved',
    image_qa_notes  = 'Manual approve: minor color cast, acceptable'
WHERE merchant_sku = 'IC-000001';
```

---

### 19.5 Шаг 3 — Генерация контента (Пайплайн C, часть 1)

**Триггер:** автоматически ночью (scheduled) или оператор запускает вручную в n8n.

**Условие:** `image_qa_status = 'approved'` AND `content_status = 'not_started'`.

**Действия оператора:**
1. Ничего — n8n запускает workflow `content_generate` автоматически по расписанию.
2. Либо: открыть n8n → вручную запустить `content_generate` для нужных SKU.
3. По завершению: n8n отправляет email с прикреплённым `review_{date}.csv`.

**Что происходит внутри (без участия оператора):**
- n8n формирует JSON с фактами о каждой иконе
- Claude API генерирует контент (sync для ≤ 10 SKU, Batches API для > 10)
- n8n валидирует output (длины полей, обязательные ключи, compliance flags)
- n8n генерирует CSV review pack и JSONL архив

---

### 19.6 Шаг 4 — Review контента (Пайплайн C, часть 2 — human gate)

**Это единственный обязательный ручной шаг в пайплайне контента.**

**CSV-файл содержит колонки:**
```
sku | amazon_title | bullet_1 | bullet_2 | bullet_3 | site_title | compliance_flags | review_action | review_notes
```

**Действия оператора:**
1. Открыть CSV в Excel / Numbers / Google Sheets.
2. Приоритет — строки с непустыми `compliance_flags` (REVIEW_* метки).
3. Проверить `amazon_title`: ключевое слово есть, длина ≤ 200 символов, Title Case.
4. Просмотреть 5 bullets: каждый начинается с пользы, не с характеристики; < 300 символов.
5. Заполнить `review_action` для каждой строки:

| Действие | Когда использовать |
|----------|--------------------|
| `APPROVE` | Всё корректно — пускать в публикацию |
| `REJECT` | Серьёзная ошибка — нужна ручная правка, без регенерации |
| `REGEN` | Контент слабый — попросить Claude сгенерировать заново |

6. При `REJECT` или `REGEN` — добавить заметку в `review_notes` (что именно не так).
7. Сохранить CSV, загрузить в папку `review-inbox/` в Drive (или через n8n-форму).
8. n8n обрабатывает файл, применяет решения, уведомляет о результате.

**После обработки:**
- `APPROVE` → `content_status = 'approved'`, данные записаны в PostgreSQL + BC metafields
- `REJECT` → `content_status = 'not_started'` (ждёт ручной правки)
- `REGEN` → `content_status = 'not_started'`, `content_version` +1, попадёт в следующий батч

---

### 19.7 Шаг 5 — Публикация в BigCommerce (Пайплайн D)

**Триггер:** автоматически после изменения `content_status → approved`.

**Оператор ничего не делает.** n8n workflow `bc-publish`:
1. Читает карточку из PostgreSQL (`content_status = 'approved'`)
2. Проверяет: bc_product_id существует? → PATCH, иначе → POST
3. Отправляет: название, описание, цена, изображения (derivative URLs), metafields
4. Получает `bc_product_id`, записывает в PostgreSQL
5. Ставит `bc_status = 'published'`
6. Уведомляет оператора: "N SKU опубликовано в BigCommerce"

**Как проверить результат:**
```sql
SELECT merchant_sku, bc_product_id, bc_status, last_synced_at
FROM icons
WHERE bc_status = 'published'
ORDER BY last_synced_at DESC
LIMIT 20;
```
Затем открыть BigCommerce → Products и убедиться, что товары видны.

---

### 19.8 Шаг 6 — Публикация на Amazon (Пайплайн E)

**Это второй обязательный ручной шаг — финальный human gate перед Amazon.**

**Условие:** `content_status = 'approved'` AND `bc_status = 'published'`.

**Действия оператора:**
1. n8n формирует список готовых к Amazon SKU, отправляет оператору сводку (email / n8n UI).
2. Оператор просматривает список — особенно `amazon_browse_node` и `amazon_price`.
3. **Подтверждает публикацию**: нажимает кнопку в n8n (или отвечает на email с командой).
4. n8n workflow `amazon-syndicate` запускается:
   - Читает Amazon payload из PostgreSQL
   - Вызывает SP-API Listings API: `PUT /listings/2021-08-01/items/{sellerId}/{sku}`
   - Получает ASIN в ответе
   - Записывает ASIN в PostgreSQL + BC metafield `catalog.source/asin`
   - Ставит `amazon_status = 'live'`
5. Оператор получает уведомление: "N SKU опубликовано на Amazon, ASINs: [список]"
6. Оператор открывает Amazon Seller Central → проверяет листинги визуально (фото, цена, категория).

**Если листинг уходит в suppressed:**
1. Открыть SC → Inventory → Fix stranded listings
2. Проверить: compliance error, price alert, image issue
3. Исправить в PostgreSQL (если данные), перезапустить `amazon-syndicate` для этого SKU
4. Если ошибка категориальная — обновить `amazon_browse_node` и regenerate

### 19.10 Обработка ошибок батча

**Как найти проблемные строки:**
```sql
SELECT bi.sku, bi.last_step, bi.error_code, bi.error_detail, bi.retry_count
FROM batch_items bi
JOIN batch_runs br ON bi.batch_id = br.batch_id
WHERE bi.status = 'FAILED'
  AND br.created_at > NOW() - INTERVAL '24 hours'
ORDER BY br.created_at DESC;
```

**Типичные ошибки и действия:**

| error_code | Что произошло | Действие |
|------------|---------------|----------|
| `BC_RATE_LIMIT` | Превышен лимит BC API | Ничего — n8n retry с backoff автоматически |
| `BC_DUPLICATE_SKU` | SKU уже существует в BC | Проверить bc_product_id в PostgreSQL, переключить на PATCH |
| `AMZN_INVALID_BROWSE_NODE` | Неверный browse node | Обновить `amazon_browse_node` в PostgreSQL → retry |
| `AMZN_PRICE_ALERT` | Цена ниже минимального порога SC | Повысить `amazon_price` → retry |
| `AMZN_IMAGE_REJECTED` | Фото не прошло модерацию Amazon | Перепроверить изображение, re-derive → retry |
| `AMZN_LISTING_INACTIVE` | Листинг создан, но неактивен | Открыть SC, проверить требования категории |
| `CONTENT_VALIDATION_ERROR` | Output Claude не прошёл валидацию | Пересмотреть промпт, regenerate SKU |

**Перезапуск после исправления:**
- Исправить данные в PostgreSQL
- В n8n: найти batch_id → Retry failed items
- batch engine подхватит только FAILED строки, не трогая успешные

### 19.11 n8n Workflow Groups — полная карта

```
Group: catalog_ops
  source-import         — загрузка CSV от поставщика → PostgreSQL

Group: image_ops
  image-ingest          — Drive webhook → stage_a_to_b → stage_b_to_c

Group: publication_ops
  bc-publish            — PostgreSQL → BC REST API v3

Group: inventory_ops

  alert-low-stock       — проверка порогов, уведомления оператору
```

### 19.12 SQL-запросы оператора (quick reference)

**Статус каталога одним взглядом:**
```sql
SELECT
  image_qa_status,
  content_status,
  bc_status,
  amazon_status,
  COUNT(*) AS cnt
FROM icons
GROUP BY 1, 2, 3, 4
ORDER BY 1, 2, 3, 4;
```

**Что готово к генерации контента:**
```sql
SELECT merchant_sku, saint_or_subject, material
FROM icons
WHERE image_qa_status = 'approved'
  AND content_status  = 'not_started';
```

**Что готово к публикации на Amazon:**
```sql
SELECT merchant_sku, amazon_title, amazon_price, amazon_browse_node
FROM icons
WHERE content_status = 'approved'
  AND bc_status      = 'published'
  AND amazon_status  = 'not_listed';
```

**Активные SKU с низким инвентарём:**
```sql
SELECT merchant_sku, saint_or_subject, site_stock, amazon_reserved,
       (site_stock + amazon_reserved) AS total
FROM icons
WHERE bc_status = 'published'
  AND (site_stock < 3 OR amazon_reserved < 3)
ORDER BY total ASC;
```

**История батчей за последние 7 дней:**
```sql
SELECT batch_type, status, item_count, success_count, failure_count,
       created_at, completed_at
FROM batch_runs
WHERE created_at > NOW() - INTERVAL '7 days'
ORDER BY created_at DESC;
```
